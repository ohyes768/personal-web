# 前端模块化开发规范

## 一、目标

确保前端各功能模块独立开发、互不影响，避免以下问题：
- 开发新模块导致原有模块被意外修改
- 代码风格不一致
- 类型定义重复
- API 调用方式混乱
- 组件无法复用

---

## 二、目录结构规范

```

│   │   │       └── [...path]/route.ts
│   │   └── modules/             # 业务模块页面

│   │   ├── hooks/             # 通用自定义 Hooks

│   │   ├── utils/             # 通用工具函数
│   │   │   └── ...

│   │       └── ...
```

### 2.2 模块命名规范

| 场景 | 命名格式 | 示例 |
|------|---------|------|
| 模块目录 | kebab-case | `douyin`, `economic`, `news` |
| 组件文件 | PascalCase | `VideoCard.tsx`, `TimeRangeSelector.tsx` |
| 工具文件 | camelCase | `dataFilterUtils.ts`, `dateCalculators.ts` |
| 类型文件 | camelCase | `api.ts` (types 放在文件内) 或 `types.ts` |
| Hook 文件 | camelCase | `useEconomicData.ts`, `useDarkMode.ts` |

### 2.3 文件数量限制

遵循全局规范：
- TypeScript/JavaScript 文件不超过 300 行
- 每层文件夹文件数不超过 8 个
- 超过限制需拆分子文件夹

## 三、API 调用规范

### 3.1 核心原则

**严禁**在页面组件中直接使用 `fetch`、`axios` 或其他 HTTP 客户端。

**必须**通过 `lib/modules/{module}/api.ts` 进行所有 API 调用。

### 3.2 API 调用层次

```
页面组件 (app/modules/{module}/page.tsx)
    ↓ 调用
模块 Hooks (lib/modules/{module}/hooks.ts)
    ↓ 调用
模块 API (lib/modules/{module}/api.ts)
    ↓ 调用
统一客户端 (lib/api/client.ts)
    ↓ 调用
后端 API
```

### 3.3 统一客户端配置

**lib/api/client.ts**（唯一入口）：

```typescript
/**
 * 统一的 API HTTP 客户端
 * 全项目唯一入口，所有 API 调用必须通过此客户端
 */
import type { ApiResponse } from './types';

class ApiClient {
  private baseUrl: string;

  constructor(baseUrl: string = '') {
    this.baseUrl = baseUrl ?? process.env.NEXT_PUBLIC_API_BASE_URL ?? 'http://localhost:3000';
  }

  private async request<T>(url: string, options?: RequestInit): Promise<T> {
    // ... 统一错误处理、请求拦截等
  }

  get<T>(endpoint: string, params?: Record<string, any>): Promise<T> { /* ... */ }
  post<T>(endpoint: string, data?: any): Promise<T> { /* ... */ }
  put<T>(endpoint: string, data?: any): Promise<T> { /* ... */ }
  delete<T>(endpoint: string): Promise<T> { /* ... */ }
}

export const apiClient = new ApiClient();
```

### 3.4 模块 API 封装

**lib/modules/{module}/api.ts**：

```typescript
/**
 * {module} 模块 API 封装
 * 所有 {module} 相关 API 调用必须通过此文件
 */
import { apiClient } from '@/lib/api/client';
import type { ModuleType1, ModuleType2 } from './types';

export const {module}Api = {
  // 命名：动词 + 名词
  getItems: (page: number = 1, pageSize: number = 20) =>
    apiClient.get<ModuleResponse>('/api/{module}/items', {
      page,
      page_size: pageSize
    }),

  getItemDetail: (id: string) =>
    apiClient.get<ModuleType1>(`/api/{module}/items/${id}`),

  createItem: (data: CreateItemDto) =>
    apiClient.post<ModuleType1>('/api/{module}/items', data),

  updateItem: (id: string, data: UpdateItemDto) =>
    apiClient.put<ModuleType1>(`/api/{module}/items/${id}`, data),

  deleteItem: (id: string) =>
    apiClient.delete<void>(`/api/{module}/items/${id}`),
};
```

### 3.5 模块 Hook 封装

**lib/modules/{module}/hooks.ts**：

```typescript
/**
 * {module} 模块自定义 Hooks
 * 封装组件中可复用的状态逻辑
 */
import { useState, useEffect } from 'react';
import { {module}Api } from './api';
import type { ModuleType1 } from './types';

export function use{Module}Items(page: number, pageSize: number) {
  const [data, setData] = useState<ModuleType1[]>([]);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);

  const fetchItems = async () => {
    setLoading(true);
    setError(null);
    try {
      const response = await {module}Api.getItems(page, pageSize);
      setData(response.items);
    } catch (err) {
      setError(err instanceof Error ? err.message : '获取数据失败');
    } finally {
      setLoading(false);
    }
  };

  useEffect(() => {
    fetchItems();
  }, [page, pageSize]);

  return { data, loading, error, refetch: fetchItems };
}
```

## 四、类型定义规范

### 4.1 类型定义层次

```
lib/api/types.ts           # 通用 API 类型（ApiResponse 等）
lib/modules/{module}/types.ts  # 模块专属类型
lib/types/*.ts            # 全局业务类型（跨模块共享）
```

### 4.2 通用 API 类型

**lib/api/types.ts**：

```typescript
/**
 * 通用 API 响应类型
 */
export interface ApiResponse<T = any> {
  success: boolean;
  data: T;
  message?: string;
  error?: string;
}

/**
 * 分页响应类型
 */
export interface PaginatedResponse<T> {
  total_count: number;
  items: T[];
  page: number;
  page_size: number;
  total_pages: number;
}

/**
 * 列表查询参数
 */
export interface ListQueryParams {
  page?: number;
  page_size?: number;
  sort_by?: string;
  sort_order?: 'asc' | 'desc';
}
```

### 4.3 模块专属类型

**lib/modules/{module}/types.ts**：

```typescript
/**
 * {module} 模块类型定义
 * 仅限本模块使用，不导出到其他模块
 */

// 实体类型
export interface {Module}Item {
  id: string;
  name: string;
  status: 'active' | 'inactive';
  created_at: string;
  updated_at: string;
}

// DTO 类型
export interface Create{Module}Dto {
  name: string;
}

export interface Update{Module}Dto {
  name?: string;
}

// 响应类型
export interface {Module}Response {
  items: {Module}Item[];
  total_count: number;
}

// 组件 Props 类型（可选）
export interface {Module}CardProps {
  item: {Module}Item;
  onClick: () => void;
}
```

### 4.4 禁止的行为

| 禁止行为 | 原因 |
|---------|------|
| 在页面组件中重新定义已存在的类型 | 导致类型重复，难以维护 |
| 在 `lib/api/types.ts` 中定义业务类型 | 应放在对应模块的 types.ts 中 |
| 使用 `any` 或未定义的 JSON | 违背强类型原则，需征求用户同意 |

## 五、组件开发规范

### 5.1 组件拆分原则

| 指标 | 限制 | 超过限制必须拆分 |
|------|------|-----------------|
| 文件行数 | 300 行 | 抽离子组件或 hooks |
| 组件复杂度 | 3 层嵌套 | 抽离子组件 |
| Props 数量 | 7 个 | 考虑使用配置对象 |
| 单个组件职责 | 单一 | 违反原则需拆分 |

### 5.2 页面组件职责

**app/modules/{module}/page.tsx** 只负责：
- 状态管理（useState、useReducer）
- 事件绑定（onClick、onChange 等）
- 布局组合（渲染子组件）

**禁止**：
- 复杂的业务逻辑（应抽离为 hooks 或服务）
- 重复的 UI 渲染（应抽离为子组件）
- 直接调用 API（应通过模块 hooks）

### 5.3 组件命名规范

| 类型 | 命名格式 | 示例 |
|------|---------|------|
| 页面组件 | `{Module}Page` | `DouyinPage`（通常通过文件名隐含） |
| UI 组件 | `{Name}` | `Button`, `Modal`, `Tabs` |
| 业务组件 | `{Module}{Name}` | `VideoCard`, `NewsItem`, `ChartContainer` |
| Hook | `use{Name}` | `useEconomicData`, `useDarkMode` |

### 5.4 组件 Props 规范

```typescript
/**
 * 组件 Props 必须定义接口
 */
interface VideoCardProps {
  // 必填属性
  video: VideoInfo;
  onPlay: (videoId: string) => void;

  // 可选属性（显式标注）
  showThumbnail?: boolean;
  className?: string;

  // 回调函数（以 on 开头）
  onClick?: () => void;
  onDelete?: (id: string) => void;
}

/**
 * 组件定义
 */
export function VideoCard({
  video,
  onPlay,
  showThumbnail = true,
  className,
  onClick,
  onDelete,
}: VideoCardProps) {
  // ...
}
```

## 六、共享 UI 组件规范

### 6.1 基础 UI 组件（components/ui/）

特点：无业务逻辑，纯展示组件

```
components/ui/
├── Button.tsx           # 按钮
├── Modal.tsx            # 弹框
├── Tabs.tsx             # Tab 切换
├── Loading.tsx          # 加载状态
├── Badge.tsx            # 徽章
├── Input.tsx            # 输入框
├── Select.tsx           # 下拉选择
└── Table.tsx            # 表格
```

### 6.2 业务模块组件（components/modules/）

特点：包含业务逻辑，可跨模块复用

```
components/modules/
├── douyin/
│   ├── VideoCard.tsx    # 视频卡片
│   └── VideoModal.tsx   # 视频详情弹框
├── economic/
│   ├── ChartContainer.tsx  # 图表容器
│   └── TimeRangeSelector.tsx # 时间范围选择器
└── news/
    └── NewsCard.tsx     # 新闻卡片
```

### 6.3 组件复用规则

| 场景 | 放置位置 |
|------|---------|
| 多个模块使用的通用 UI | `components/ui/` |
| 某个模块内部复用 | `components/modules/{module}/` |
| 仅单个页面使用 | 与页面同目录的 `components/` |

## 八、状态管理规范

### 8.1 状态管理层次

| 状态类型 | 管理方式 | 示例 |
|---------|---------|------|
| 组件本地状态 | useState | 表单输入、UI 交互 |
| 跨组件共享状态 | Context API | 主题、用户信息 |
| 服务端状态 | React Query / 自定义 Hook | API 数据、缓存 |
| 复杂应用状态 | Zustand / Redux | 全局状态（如有需要） |

### 8.2 API 数据获取

**优先使用自定义 Hook**：

```typescript
// lib/modules/{module}/hooks.ts
export function use{Module}Data() {
  const [data, setData] = useState<ModuleData | null>(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);

  const fetchData = async () => {
    setLoading(true);
    try {
      const result = await {module}Api.getData();
      setData(result);
    } catch (err) {
      setError(err instanceof Error ? err.message : '获取失败');
    } finally {
      setLoading(false);
    }
  };

  // ...
}
```

## 九、代码质量规范

### 9.1 强制规则

| 规则 | 说明 |
|------|------|
| TypeScript 强类型 | 禁止使用 `any`，特殊情况需征求用户同意 |
| 无循环依赖 | 模块之间单向依赖，禁止 A 依赖 B，B 又依赖 A |
| 单一职责 | 每个函数/组件只做一件事 |
| 函数参数 | 不超过 5 个，超过使用对象参数 |
| 文件行数 | TS/JS 不超过 300 行，静态语言不超过 400 行 |

### 9.2 ESLint / Prettier 配置

统一配置，确保代码风格一致：

```json
{
  "extends": [
    "next/core-web-vitals",
    "prettier"
  ],
  "rules": {
    "@typescript-eslint/no-explicit-any": "error",
    "@typescript-eslint/explicit-module-boundary-types": "error",
    "no-console": ["warn", { "allow": ["warn", "error"] }],
    "prefer-const": "error"
  }
}
```

## 十、模块独立性检查清单

开发新模块时，必须确保：

| 检查项 | 通过标准 |
|--------|---------|
| API 调用 | 全部通过 `lib/modules/{module}/api.ts` |
| 类型定义 | 无重复，全部在 `lib/modules/{module}/types.ts` |
| 组件拆分 | 单文件不超过 300 行 |
| 状态管理 | 逻辑抽离为 hooks |
| 共享资源 | 使用共享 UI 组件，不重复开发 |
| 环境变量 | 不硬编码，使用 `.env.local` |
| 导入路径 | 使用 `@/` 别名，相对路径仅限同级 |
| 依赖模块 | 无反向依赖（新模块不依赖更高级模块） |

## 十一、重构检查清单

重构现有模块时，检查并修复：

| 问题 | 检查方式 | 修复方法 |
|------|---------|---------|
| 直接使用 fetch | 搜索 `fetch(` | 替换为模块 API |
| 类型定义重复 | 搜索 `interface {Type}` | 删除重复，统一管理 |
| 单文件过长 | 检查行数 | 拆分子组件或 hooks |
| 硬编码 URL | 搜索 `http://` | 使用环境变量 |
| 循环依赖 | 手动检查或工具检测 | 重构模块结构 |
| 缺少类型标注 | 搜索 `any` | 添加明确类型 |

## 十二、开发流程规范

### 12.1 新增模块流程

1. 创建模块目录 `app/modules/{module}/`
2. 创建模块 API `lib/modules/{module}/api.ts`
3. 创建模块类型 `lib/modules/{module}/types.ts`
4. 创建模块 Hooks `lib/modules/{module}/hooks.ts`
5. 创建页面组件 `app/modules/{module}/page.tsx`
6. 如有需要，创建业务组件 `components/modules/{module}/`
7. 添加 API 路由代理 `app/api/{module}/[...path]/route.ts`
8. 更新文档

### 12.2 修改现有模块流程

1. 识别影响范围（API、类型、组件）
2. 仅修改必要文件
3. 避免影响其他模块
4. 运行测试确认
5. 更新相关文档
